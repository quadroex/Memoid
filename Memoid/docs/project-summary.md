# Memoid: опис проєкту

## Тема проєкту

Memoid - вебзастосунок для створення мемів на основі готових шаблонів і власних зображень користувача.

## Мета

Мета проєкту - реалізувати навчальний ASP.NET Core Web API застосунок із базою даних PostgreSQL, статичним frontend та browser-based редактором мемів на HTML5 Canvas.

## Предметна область

Предметна область охоплює створення, збереження та перегляд мемів. Користувач може створювати категорії, додавати шаблони, завантажувати власні зображення, накладати текст, застосовувати прості ефекти й зберігати результат у галерею.

## Актори та основні сценарії

Основний актор - користувач вебзастосунку.

Користувач може:

- створити категорію шаблонів;
- додати шаблон мема із зображенням;
- вибрати шаблон у редакторі;
- завантажити власне зображення;
- додати верхній і нижній текст;
- змінити шрифт, розмір, колір і позицію тексту;
- застосувати ефекти зображення;
- застосувати шакалізацію;
- вибрати режим кадрування;
- завантажити готовий мем;
- зберегти мем у галерею;
- переглянути, завантажити, додати в улюблені або видалити створений мем.

## Основні сутності

### MemeCategory

Категорія для групування шаблонів мемів. Містить назву, опис, дату створення, ознаку активності та список шаблонів.

### MemeTemplate

Шаблон мема. Містить назву, шлях до зображення, посилання на категорію, дату створення, ознаку активності та зв'язок зі створеними мемами.

### GeneratedMeme

Створений мем. Містить назву, шлях до готового зображення, тип джерела, посилання на шаблон або власне зображення, текстові налаштування, ефект, ознаку улюбленого та дату створення.

## Frontend і backend

Frontend реалізовано як статичні файли у `wwwroot`: HTML, CSS і JavaScript. Він відповідає за інтерфейс користувача, завантаження даних через `fetch`, роботу canvas-редактора, накладання тексту, ефекти та формування готового зображення.

Backend реалізовано як ASP.NET Core Web API. Він відповідає за HTTP endpoints, валідацію даних, збереження файлів у `wwwroot/uploads`, роботу з EF Core та збереження метаданих у PostgreSQL.

## Чому Web API, а не MVC

У цьому проєкті використано Web API, тому що frontend і backend розділені. Backend повертає JSON і приймає HTTP-запити, а frontend самостійно відображає сторінку та взаємодіє з API через JavaScript `fetch`. Це відповідає сучасному підходу до розділення клієнтської та серверної частини.

## Реалізовані use cases

- перегляд категорій;
- створення, редагування та деактивація або видалення категорій;
- перегляд шаблонів;
- створення шаблону із завантаженням зображення;
- вибір шаблону в редакторі;
- завантаження власного зображення;
- створення мема на canvas;
- налаштування тексту;
- застосування ефектів;
- застосування шакалізації;
- вибір режиму кадрування;
- збереження готового мема;
- перегляд галереї;
- повноекранний перегляд мема;
- завантаження мема;
- перемикання улюбленого статусу;
- фільтр улюблених;
- видалення запису створеного мема.

## Тестування

Для перевірки проєкту використовуються:

- запуск головної сторінки;
- перевірка OpenAPI JSON;
- ручні HTTP-запити до API;
- тестування форм у браузері;
- перевірка завантаження файлів;
- перевірка canvas-редактора;
- перевірка галереї.

Детальний список ручних тестів наведено у `docs/manual-test-checklist.md`.
